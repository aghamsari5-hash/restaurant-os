# RestaurantOS UI Rules

Version: 1.0

---

# Goal

The user interface must be:

Simple

Fast

Beautiful

Consistent

Easy to Learn

AI Friendly

---

# Design Philosophy

The interface should feel similar to Snapp.

Not because of its colors,

but because of its simplicity.

The user should understand every screen within a few seconds.

---

# Core Principles

Less Clicks

Less Text

Less Colors

Less Popups

Less Complexity

More Speed

More Clarity

---

# User Experience

Every action should feel immediate.

Users should never wonder:

"What should I click next?"

The interface must guide them naturally.

---

# Screen Rules

Every screen has:

Title

Search

Primary Action

Content

Optional Filters

Nothing else.

---

# Maximum Elements

Dashboard Cards

Maximum 8

Main Menu

Maximum 10

Primary Buttons

Maximum 2

Floating Buttons

Maximum 1

---

# Colors

Primary Color

Brand Color

Success

Green

Warning

Orange

Danger

Red

Information

Blue

Neutral

Gray

Avoid unnecessary colors.

---

# Typography

Maximum

3 Font Sizes

Maximum

2 Font Weights

No decorative fonts.

Readable only.

---

# Spacing

Consistent spacing everywhere.

No crowded screens.

Large touch targets.

---

# Buttons

Every button must clearly explain its action.

Good

Create Order

Bad

Submit

---

# Icons

Every important action has an icon.

Never use icons alone.

Icons + Text

---

# Forms

Minimal fields.

Group related fields.

Validate immediately.

Show clear errors.

---

# Validation

Show errors beside fields.

Never show generic errors.

Good

Phone Number is invalid.

Bad

Something went wrong.

---

# Tables

Search

Filter

Sort

Pagination

Column Visibility

Export

Responsive

Every table follows the same behavior.

---

# Dashboard

Simple.

Important information first.

Charts only when useful.

No decorative widgets.

---

# Navigation

Left Sidebar

Desktop

Bottom Navigation

Mobile

Maximum Navigation Depth

3 Levels

---

# Search

Global Search

Available everywhere.

Search must be fast.

---

# Empty States

Every empty page explains:

Why it is empty.

How to add data.

Provide one action button.

Example

No Products Yet

Create Product

---

# Loading

Skeleton Loading

Preferred.

Avoid full-screen spinners.

---

# Notifications

Small

Clear

Short

Success

Warning

Error

Info

Maximum Display Time

5 Seconds

---

# Confirmation

Require confirmation only for dangerous actions.

Examples

Delete

Refund

Cancel Order

Never confirm simple actions.

---

# Responsive Design

Desktop

Tablet

Mobile

Must all be supported.

---

# Accessibility

Keyboard Navigation

Screen Reader Friendly

High Contrast

Large Click Area

Visible Focus States

---

# Images

Optimized

Lazy Loaded

WebP Preferred

Thumbnail First

---

# Animations

Fast

Subtle

Never delay user actions.

Animation Duration

Maximum 200ms

---

# API Rules

One page

Maximum 5 API Requests

Reuse existing data whenever possible.

Avoid duplicate requests.

---

# Performance

Screen Load

Target

< 2 Seconds

Button Response

< 100ms

Search

< 300ms

---

# Feature Visibility

Disabled Features

Must completely disappear.

Do not show disabled menus.

Do not show disabled buttons.

Do not show disabled pages.

---

# Menu Rules

Orders

POS

Customers

----------------

Products

Inventory

----------------

Reports

Accounting

# Error Pages

404

Friendly

500

Friendly

Offline

Helpful

# Mobile Rules

Thumb Friendly

Bottom Actions

Large Buttons

QR Support

---

# POS Rules

Large Buttons

Fast Search

Keyboard Shortcuts

Touch Friendly

Minimum Clicks

---

# Dashboard Rules

Show only important KPIs.

Today's Sales

Orders

Customers

Top Products

Low Stock

Pending Orders

No unnecessary charts.

---

# Design Consistency

Every page follows the same layout.

Users should never relearn the interface.

---

# AI Rules

Claude Code must never:

Create crowded pages.

Add unnecessary dialogs.

Use more than 5 API calls.

Invent new UI patterns.

Use inconsistent spacing.

Change design style between pages.

Every new page must follow this document.

---

# Final Rule

The best interface is the one that users never need training for.

RestaurantOS should feel natural from the first minute.
